# Phase 0 Research: Enrolment Contracts

Each decision below resolves a technical unknown from the Technical Context. Format: Decision / Rationale / Alternatives considered.

## R1. Contracted weekdays and consent are JSONB owned collections/types on `Contract`, not separate tables

**Decision**: `Contract.ContractedDays` is an EF Core owned collection (`ContractedDay { DayOfWeek Weekday, TimeOnly StartTime, TimeOnly EndTime }`) mapped via `OwnsMany(...).ToJson("contracted_days")`; `Contract.Consent` is a single owned type (`ContractConsent { PhotosInternal, PhotosWebsite, PhotosSocialMedia, VideoInternal, PhotosPress }`, all `bool`) mapped via `OwnsOne(...).ToJson("consent")`. Both live as JSONB columns on the single `contracts` table — no `contract_days` or `contract_consent` join tables.

**Rationale**: The feature description explicitly asks for consent as JSONB ("Replaces the old single photo_consent boolean"); extending the same JSONB-owned-type approach to contracted days keeps both compound fields on one row, avoids two extra tables for data that is always read/written as a whole alongside its parent contract, and mirrors the constitution's own precedent for `child_events` ("live in a single JSONB-backed table — do not create a separate table per event type"). EF Core 10 / Npgsql supports `ToJson()` on owned collections and owned types natively, so no custom value converter or raw SQL is needed. The day-overlap validator (FR-005/006) only ever needs a child's small number of *active* contracts loaded into memory to compare weekdays — it never needs to query "all contracts with weekday X" at the SQL level, so there is no relational-querying need that JSONB would block.

**Alternatives considered**: A separate `ContractDay` table (`ContractId`, `Weekday`, `StartTime`, `EndTime`) — rejected as unnecessary relational normalization for data with no independent identity, no cross-contract queries, and a hard upper bound of 5 rows (Mon–Fri) per contract; a single free-text/JSON blob with no EF-level shape (raw `string` column, manually serialized) — rejected in favor of `ToJson()`'s typed owned-entity mapping, which gives compile-time safety and change-tracking without hand-rolled (de)serialization.

## R2. Contract activation concurrency is serialized via a Postgres advisory lock keyed on the child, reusing the pattern (not the code) from the org-registration race fix

**Decision**: A new `IAdvisoryLockService.RunExclusiveAsync<T>(Guid key, Func<Task<T>> action, CancellationToken)` port, implemented by `PostgresAdvisoryLockService` in `ChildCare.Infrastructure/Concurrency/` using the same `pg_advisory_lock`/`pg_advisory_unlock` technique as `TenantProvisioningService.RunExclusiveAsync` (feature 001, commit `4625480`). `ActivateContractCommandHandler` wraps its entire check-then-write (one-active-per-location check, day-overlap check, status transition) in `RunExclusiveAsync(childId, ...)`, so two concurrent activation attempts for the same child are fully serialized — the second one only proceeds after the first has committed or rolled back, and therefore always observes the first's result.

**Correction found during implementation**: `ContractActivationChecker`'s conflict query (`db.Contracts.Where(c => ... && c.Status == ContractStatus.Active)`) is translated to SQL and evaluated against the *persisted* column value — it is not aware of an unsaved in-memory property change on an already-tracked entity. An early draft of `AmendContractCommandHandler` assumed that setting the predecessor's `Status = Ended` in-memory before this query ran would be enough for EF Core's identity-map to exclude it from the result set; `AmendContract_WithFutureEffectiveDate_EndsOriginalAndActivatesSuccessor` failed immediately (`errors.contract.already_active_at_location` on a plain same-location rate amendment), proving that assumption wrong — the predecessor's row still satisfies `Status == Active` in the database and is returned regardless of its pending in-memory change. Fixed by adding an explicit `excludeContractId` parameter to `ContractActivationChecker.CheckAndActivateAsync`, which `AmendContractCommandHandler` passes as the predecessor's id.

**Rationale**: FR-006 requires the one-active-per-location and day-overlap checks to be atomic under concurrency — exactly the class of bug the org-registration fix discovered (a check-then-write race where the loser doesn't see the winner's write in time). That fix is proven, tested (20 isolated reruns + full-suite runs), and needs no new infrastructure (Postgres advisory locks are session-scoped and already available on the same connection string). A **new**, feature-scoped `IAdvisoryLockService` port is introduced rather than promoting `ITenantProvisioningService.RunExclusiveAsync` into a shared abstraction — that method lives on a feature-001-specific interface, and generalizing it would mean modifying feature 001's already-shipped, tested code purely for this feature's benefit, which is an unrequested refactor outside this feature's scope (the user's standing instruction is to suggest refactors, never apply them unprompted). The lock key is the `childId` (not the location, not the contract) because the invariant being protected — "this child's active contracts never share a weekday, anywhere" — is scoped to the child, matching how FR-005 itself is phrased.

**Alternatives considered**: `SERIALIZABLE` transaction isolation with retry-on-conflict — rejected as more complex to reason about and test than an explicit lock (requires catching Postgres serialization-failure error codes and retrying, whereas the advisory lock simply makes the second caller wait, with no retry loop needed); a unique database constraint encoding "no two active contracts for the same child share a weekday" — rejected as impractical to express as a single SQL constraint given JSONB-stored weekdays and the cross-location aspect of the rule; promoting `RunExclusiveAsync` to a shared interface — rejected per the scope reasoning above (tracked as a "worth doing later, not now" note rather than actioned here).

## R3. `ILocationDeactivationGuard` and `IChildDeactivationGuard` are both implemented here, checking for any `active` contract

**Decision**: `ContractLocationDeactivationGuard : ILocationDeactivationGuard` and `ContractChildDeactivationGuard : IChildDeactivationGuard` are both added, each querying `db.Contracts.AnyAsync(c => c.LocationId == id && c.Status == ContractStatus.Active)` / `c.ChildId == id && c.Status == ContractStatus.Active` respectively. Both are registered in `Program.cs` via `services.AddScoped<ILocationDeactivationGuard, ContractLocationDeactivationGuard>()` and `services.AddScoped<IChildDeactivationGuard, ContractChildDeactivationGuard>()`, additively — `DeactivateLocationCommandHandler` and `DeactivateChildCommandHandler` already resolve `IEnumerable<...>`, so this feature's guard joins whatever else is registered (e.g., feature 005's future `IStaffDeactivationGuard`-style additions to `ILocationDeactivationGuard`) without overwriting anything.

**Rationale**: This is exactly the extension point features 004 (`ILocationDeactivationGuard`) and 006 (`IChildDeactivationGuard`) reserved and documented as "feature 007 registers its own once a child/location can have an active contract." Both existing doc comments name this feature explicitly.

**Alternatives considered**: None seriously considered beyond the established pattern — both interfaces already exist with this exact contract.

## R4. Contract PDF generation via a new `IContractPdfGenerator` port, QuestPDF adapter

**Decision**: `IContractPdfGenerator.GenerateAsync(ContractPdfModel model, CancellationToken)` returning `byte[]`, defined in `ChildCare.Application/Common/`, implemented by `QuestPdfContractGenerator` in `ChildCare.Infrastructure/Pdf/` using the QuestPDF fluent document API (constitution's fixed PDF library). `QuestPDF.Settings.License = QuestPDF.Infrastructure.LicenseType.Community;` is set once at startup in `Program.cs` (required by QuestPDF before generating any document; Community licence is free and applies here — solo/small-team, pre-revenue). `GenerateContractPdfQueryHandler` loads the contract (and its child/location names) and calls this port; `ContractsEndpoints.cs` returns the bytes as `Results.File(bytes, "application/pdf")`.

**Rationale**: Mirrors the `IProfilePhotoStorage`/`GcsProfilePhotoStorage` port/adapter split already established for external-service seams (features 005/006) — `ChildCare.Application` depends only on the interface, keeping QuestPDF itself (and any future PDF library swap) confined to `ChildCare.Infrastructure`, consistent with constitution Principle VII (`ChildCare.Application` stays framework-agnostic). This is also the first QuestPDF usage in the codebase (constitution names it as the fixed Phase 1 PDF library but no prior feature needed a PDF yet), so the package reference is added to `ChildCare.Infrastructure.csproj` here.

**Alternatives considered**: Generating the PDF directly inside the MediatR handler with no port — rejected as it would leak a third-party library into `ChildCare.Application`, breaking the pattern every other external-service dependency in this codebase already follows; a different PDF library — rejected, QuestPDF is fixed by the constitution's Technology Stack Constraints section.

## R5. Amendment and termination are distinct commands, not a single "update contract" operation

**Decision**: `AmendContractCommand` (ends the current contract, creates+activates a successor with new terms — FR-007/008) and `TerminateContractCommand` (ends the current contract, no successor — FR-009a) are two separate MediatR commands, alongside `UpdateContractCommand` (edits an untouched `Draft` contract's terms in place, no versioning — only valid while `Draft`) and `ActivateContractCommand` (transitions `Draft` → `Active`, FR-003/004/005/006).

**Rationale**: These are semantically different operations with different validation and side effects (amendment always produces and activates a new row subject to the full overlap check excluding the contract being ended; termination never produces a new row; draft-update never touches status at all) — collapsing them into one endpoint/command would require a discriminator field and branching logic inside a single handler, which is harder to test and reason about than four small, single-purpose commands. This also matches the existing codebase convention of one MediatR command per distinct user action (e.g., features 004–006 each have separate `Deactivate`/`Reactivate` commands rather than one "SetStatus" command).

**Alternatives considered**: A single `UpdateContractCommand` handling both in-place draft edits and post-activation amendments via a status check inside the handler — rejected as conflating two different audit-trail guarantees (draft edits are mutable-in-place by design; active contracts must never be mutated in place, only superseded) into one code path, risking an accidental mutation of an active contract's terms.

## R6. `tarief_code`/`rate_valid_until` are plain nullable columns, added now but unused

**Decision**: `Contract.TariefCode` (`string?`) and `Contract.RateValidUntil` (`DateOnly?`) are added to the entity and migration now, both always `null` in this feature (never set by any command, never read by any handler, not exposed on `ContractResponse`).

**Rationale**: FR-013 explicitly requires this so Phase 3's IKT subsidy-rate support can be added without a schema migration — directly following the feature description's own instruction, and consistent with the constitution's phase-scoping discipline (Phase 3 fields must exist as placeholders, not be built out ahead of schedule).

**Alternatives considered**: Omitting the columns entirely and adding them in a Phase 3 migration — rejected because the feature description explicitly asks for them now, precisely to avoid a future migration against a populated `contracts` table.
