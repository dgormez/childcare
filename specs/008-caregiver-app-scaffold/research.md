# Phase 0 Research: Caregiver App Scaffold

Each decision below resolves a technical unknown from the Technical Context. Format: Decision / Rationale / Alternatives considered.

## R1. Offline queue and read cache are two plain SQLite tables, managed by hand-written CRUD, not an ORM

**Decision**: `offline_queue` and `read_cache` (exact schema per the feature description) are created via `expo-sqlite`'s existing `initDb()`/migration pattern (`mobile/services/localDb.ts`, already used for the Habits skeleton's local tables). Two small service modules (`offlineQueue.ts`, `readCache.ts`) provide typed CRUD functions over them — no ORM, no schema-migration library beyond the `CREATE TABLE IF NOT EXISTS` pattern `localDb.ts` already follows.

**Rationale**: `expo-sqlite` is already a dependency and already has an established local-table pattern in this codebase (the Habits skeleton's `config` table, read by `tryRestoreSession()`) — extending that pattern is simpler than introducing a new persistence library for two small, well-understood tables. Constitution Principle VII (monolith-first, no premature abstraction) argues against pulling in a mobile ORM for two tables.

**Alternatives considered**: `expo-sqlite/kv-store` (key-value only) — rejected, `offline_queue` needs to be queried/ordered by `created_at`, which a pure KV store can't do without deserializing every value; a dedicated offline-sync library (e.g., WatermelonDB) — rejected as significant new complexity/learning surface for a generic queue this feature can implement directly in a few hundred lines, consistent with "no new abstraction until a second consumer needs it" (features 009/010 are the second and third consumers, and they only need to call `enqueue()`/register a handler, not touch the library's own model layer).

## R2. API client: openapi-typescript (dev-time codegen) + openapi-fetch (runtime), generated against the backend's dev OpenAPI document

**Decision**: A new `mobile/scripts/generate-api-client.sh` (or an npm script) runs `openapi-typescript http://localhost:5000/openapi/v1.json -o services/generated/api-types.ts` against a locally running backend in Development (where `MapOpenApi()` is exempted from `TenantMiddleware`'s anonymous-401 rejection — confirmed via `app.Environment.IsDevelopment()` gating in `Program.cs`). `services/apiClient.ts` wraps `openapi-fetch`'s `createClient<paths>()` with a `middleware` that attaches the Bearer token and retries once on 401 after a silent refresh, replacing the hand-rolled `request<T>()` wrapper in the old `services/api.ts`.

**Rationale**: This is the pattern PROJECT-BRIEF.md already names as decided ("Client generation: openapi-typescript + openapi-fetch — no NSwag") but that neither existing client (mobile's Habits skeleton nor web) has actually adopted yet — this feature is the first to do so, establishing the real convention rather than leaving it as an undelivered decision. Generating from the dev OpenAPI document (rather than hand-maintaining types) keeps the mobile client's request/response shapes mechanically in sync with the backend contracts documented in `backend/ERROR_KEYS.md` and each feature's `contracts/*.md`.

**Alternatives considered**: Keep the hand-rolled `fetch()` wrapper (as both existing clients do today) — rejected, this is exactly the undelivered decision this feature is meant to finally implement; Axios — rejected, explicitly rejected in PROJECT-BRIEF.md in favor of the smaller `openapi-fetch`.

## R3. Login requires an explicit organisation field; `AuthResponse`/`auth.ts` are updated to the real feature-003 contract

**Decision**: The login screen collects `organisationSlug`, `email`, `password` (all three, matching `LoginRequest` exactly) — a director communicates the slug to caregivers when provisioning their account, mirroring how a workspace/org identifier is typically shared out-of-band in multi-tenant B2B apps. `mobile/types/index.ts`'s `AuthResponse`/`AuthenticatedUser` types and `services/auth.ts` are updated to the actual current backend shape (`{ accessToken, refreshToken, user: { id, email, emailVerified, role } }`), including the previously-missing `role` field and dropping the call to `POST /api/auth/register`, which no longer exists on the backend (confirmed 404 per `specs/003-auth/contracts/auth-api.md`) — caregiver accounts are director-provisioned only (feature 005), never self-registered.

**Rationale**: No existing client (web or mobile) has actually migrated to the real feature-003 auth contract yet — both were left on a stale pre-003 shape (discovered during this feature's research, not assumed). Since the caregiver app is the app actually being built as a real, working client in this session, it must talk to the contract that exists, not the one the Habits skeleton assumed. An explicit slug field is the only option with no further dependency: no email-to-organisation lookup endpoint exists anywhere in the backend, and building one is out of scope for a "scaffold" feature.

**Alternatives considered**: Build a new backend "find my organisation by email" endpoint — rejected as unnecessary scope growth for this feature (a caregiver only ever belongs to one organisation, and a director already has an established channel — the invitation — to communicate the slug); a subdomain-per-tenant URL scheme (e.g., `myorg.childcare.app`) resolving the slug automatically — rejected, no such infrastructure exists in this project (single API host, `organisationSlug` is a request field, not a routing concern) and introducing it is a much larger, unrelated infrastructure change.

## R4. The sync engine ships as generic infrastructure with zero registered entity handlers; its own tests use a synthetic entity type

**Decision**: `syncEngine.ts` exposes `registerSyncHandler(entityType, handler)` and `syncPendingQueue()`. This feature registers none — `__tests__/services/syncEngine.test.ts` registers a synthetic `'_test_entity'` handler purely to exercise the generic queue-ordering/retry/conflict-default mechanics described in FR-012 through FR-015a, then unregisters it. Features 009/010 will each call `registerSyncHandler('child_event', ...)` / `registerSyncHandler('attendance_record', ...)` respectively when they ship.

**Rationale**: Directly matches the spec's own framing (US3: "ships no user-facing action of its own yet, but its correctness is what everything after it depends on") and the feature description's explicit "no entity_type handlers are registered yet." A synthetic test entity is the only way to test genuinely generic infrastructure before any real consumer exists — this mirrors how `IChildDeactivationGuard`/`ILocationDeactivationGuard` shipped with zero registered implementations in features 004/006, tested only via the guard interface's own contract, not a real registered guard.

**Alternatives considered**: Delay writing sync-engine tests until feature 009 registers a real handler — rejected, this would leave the single most safety-critical piece of this feature (per spec.md's own "everything after it depends on" framing) untested at the point it ships, violating constitution Principle V's real-infrastructure testing expectation applied to this feature's own scope.

## R5. Network status: `@react-native-community/netinfo`, subscribed once at the app root, exposed via a hook

**Decision**: `useNetworkStatus()` wraps `NetInfo.addEventListener`, exposing `{ isConnected: boolean }`. The offline banner (app-wide, rendered from `app/(app)/_layout.tsx`) and the sync engine's "reconnect" trigger (FR-012a) both consume this same hook/listener rather than each polling connectivity independently.

**Rationale**: `@react-native-community/netinfo` is the standard, actively-maintained React Native network-state library (used by Expo's own documentation for this exact purpose) and is not yet a dependency anywhere in this codebase, so it is the first genuinely new piece of infrastructure this feature introduces, not a replacement of an existing pattern.

**Alternatives considered**: Polling a lightweight health-check endpoint on an interval to infer connectivity — rejected, explicitly contradicts FR-012a's "no fixed-interval background polling" requirement and is strictly worse than a native OS-level connectivity listener (slower to detect changes, wastes battery/network for no benefit).

## R6. Backend read-access: a new self-service query plus two existing queries gain caller-aware scoping, not a new authorization policy

**Decision**: `GET /api/staff/me` is a new, standalone route (`StaffOrDirector` policy) outside the existing `/api/staff` `DirectorOnly` group — ASP.NET Core composes multiple `RequireAuthorization` calls on the same route additively (AND, not override), so a more permissive per-route policy cannot be layered inside a stricter group-level one; it must be its own top-level route registration, exactly like `AuthEndpoints.cs`'s existing mix of exempt/authenticated routes at different policy levels. It resolves the caller's own `StaffProfile` via the JWT's `ClaimTypes.NameIdentifier` claim (matching `AuthEndpoints.cs`'s existing claim-extraction precedent) and returns their `StaffProfileId` plus `EligibleLocationIds` (from `StaffLocationEligibility`, feature 005). `GET /api/children` (extended with an optional `groupId` filter, joining the existing `ChildGroupAssignment` table) and `GET /api/groups` are each split into their own `StaffOrDirector`-authorized route group (separate from the existing `DirectorOnly` write-route groups in the same file), with their MediatR queries gaining optional caller-identity parameters: when the caller's role is `Staff`, results are filtered to locations in their `StaffLocationEligibility`; when `Director`, behavior is byte-for-byte unchanged from features 004/006.

**Rationale**: The existing `StaffOrDirector` authorization policy (feature 003, `research.md R5`) already exists and already reads the standard `ClaimTypes.Role` claim — no new policy needs to be defined, only applied where it wasn't yet. Reusing the `GET /api/children?groupId=` shape the feature description itself already assumes (rather than inventing a different endpoint) keeps the mobile client's expectations and the backend's actual contract aligned. Location-scoping only for the `Staff` role (leaving `Director` untouched) is additive and non-breaking — every existing feature 004/006 test asserting a director's unfiltered view continues to hold.

**Alternatives considered**: A generic "list children in group X" endpoint under `/api/groups/{id}/children` instead of extending `GET /api/children` — rejected in favor of the shape the feature description already specified, avoiding a second, parallel way to list children; giving caregivers full unscoped read access (no location filtering) — rejected as an unnecessary and avoidable data-boundary weakening for a multi-location organisation (a caregiver at Location A has no legitimate need to see Location B's children).
