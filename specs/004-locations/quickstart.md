# Quickstart: Location Management

## Prerequisites

- Docker Postgres running locally
- Two organisations registered via feature 001's `POST /api/organisations/register` (as in features 002/003's quickstarts) — Org A (slug `org-a`) and Org B (slug `org-b`), each with a director account and a valid access token
- `dotnet build backend/ChildCare.sln` succeeds

## Scenario 1 — Create and manage locations (User Story 1, SC-001)

1. As Org A's director, `POST /api/locations` with `name`/`address`/`phone`/`email`/`maxCapacity` → `201`, response includes a new `id`.
2. `GET /api/locations` → the new location appears, `deactivatedAt: null`.
3. Create a second location in Org A → `GET /api/locations` returns both, independently.
4. `PUT /api/locations/{id}` changing `name`/`address` → `200`, `GET /api/locations/{id}` reflects the change.
5. `POST /api/locations` omitting `email` → `422 errors.validation`, `fieldErrors.email` populated.
6. `POST /api/locations` with `maxCapacity: 0` → `422 errors.validation`.

## Scenario 2 — Opgroeien settings are optional at creation (User Story 2, SC-002)

1. Create a location with only the core fields (Scenario 1) → confirm `naamLocatie`, `dossiernummer`, `verantwoordelijke` are `null` and `flexPermission`/`boPermission` are `false` in the response.
2. `PUT /api/locations/{id}` adding `dossiernummer`/`verantwoordelijke`/`naamLocatie` and setting `flexPermission: true` → `200`, `GET /api/locations/{id}` reflects all four new values.

## Scenario 3 — Deactivate and reactivate (User Story 3, SC-005)

1. `POST /api/locations/{id}/deactivate` on a location with no dependents → `200`, `deactivatedAt` is set.
2. `GET /api/locations` (default) → the deactivated location is absent.
3. `GET /api/locations?includeDeactivated=true` → the deactivated location is present with its `deactivatedAt` populated.
4. `POST /api/locations/{id}/reactivate` → `200`, `deactivatedAt` is `null` again; `GET /api/locations` (default) includes it again with all prior field values intact.
5. Deactivate every location in Org A (down to zero active) → no error at any step (FR-016).

## Scenario 4 — Duplicate a location (User Story 4, SC-006)

1. Create a fully-populated location (core + Opgroeien fields) in Org A.
2. `POST /api/locations/{id}/duplicate` → `201`, new `id`, all field values match the source.
3. `PUT /api/locations/{newId}` changing only `name`/`address` → `200`; `GET /api/locations/{id}` (the original) is unchanged.
4. Confirm the duplicated location's response contains no field referencing the source location's id.

## Scenario 5 — Tenant isolation (constitution Principle I, FR-007, SC-004)

1. Note a location `id` created in Org A (Scenario 1).
2. As Org B's director (different access token, different `tenant_id` claim), `GET /api/locations/{orgALocationId}` → `404 errors.location.not_found` (not `403` — the row is structurally invisible via `search_path`, indistinguishable from a genuinely nonexistent id).
3. `GET /api/locations` as Org B → Org A's locations never appear.
