# Quickstart: Authentication & Role-Based Authorization

## Prerequisites

- Docker Postgres running locally
- Two organisations registered via feature 001's `POST /api/organisations/register` (as in feature 002's quickstart) — Org A (slug `org-a`) and Org B (slug `org-b`), each with a director account
- `dotnet build backend/ChildCare.sln` succeeds

## Scenario 1 — Slug-based tenant resolution replaces the default-tenant shim (User Story 1, SC-001)

1. In Org A's schema, directly insert (or seed via a test helper) a second `TenantUser` with the same email as Org B's director but a different password.
2. `POST /api/auth/login` with `organisationSlug: "org-a"` and that shared email/Org A's password → 200, `tenant_id` claim decodes to Org A's id.
3. `POST /api/auth/login` with `organisationSlug: "org-b"` and the same email/Org B's password → 200, `tenant_id` claim decodes to Org B's id.
4. `POST /api/auth/login` with an unknown `organisationSlug` → `404 errors.auth.organisation_not_found`.
5. `POST /api/auth/login` omitting `organisationSlug` entirely → request rejected by validation, no `TenantUser` lookup attempted against any schema.

## Scenario 2 — OAuth sign-in links only, never creates (User Story 2, FR-009 fix)

1. `POST /api/auth/google` with `organisationSlug: "org-a"` and a valid Google ID token whose email matches no `TenantUser` in Org A → `401 errors.auth.invalid_credentials`; confirm no new row was created in Org A's `users` table.
2. Seed a `TenantUser` in Org A with that same email (password-based, unlinked). Repeat the same Google sign-in call → 200, and confirm that row's `GoogleId` is now set (linked, not duplicated).
3. Repeat for Apple with a JWKS-signed identity token.

## Scenario 3 — Role-based policies (User Story 3, SC-002)

1. Seed one `TenantUser` per role (Director/Staff/Parent) in Org A.
2. Call a `DirectorOnly`-protected test endpoint with each role's access token → 200 for Director, 403 for Staff and Parent.
3. Call a `StaffOrDirector`-protected test endpoint → 200 for Director and Staff, 403 for Parent.
4. Call a `ParentOnly`-protected test endpoint → 200 for Parent only.
5. Craft (test-only) a JWT with a role claim value that matches none of the three (`"unknown"`) → 403 against all three protected endpoints.

## Scenario 4 — Per-role sign-in method enforcement (FR-017)

1. Seed a `TenantUser` with `Role = Staff` and a linked Google identity (bypassing the normal flow, for test setup only — Staff accounts should never actually reach this state via the app).
2. `POST /api/auth/google` with that account's identity → `403 errors.auth.method_not_allowed_for_role`, even though the token itself is valid and matches an existing account.

## Scenario 5 — Per-device sessions unaffected (User Story 4, SC-004 — regression check)

1. Log in twice with the same account/organisation from two simulated devices.
2. Log out with the first device's refresh token → confirm the second device's token still refreshes successfully.
3. `POST /api/auth/reset-password` for that account → confirm both devices' refresh tokens are subsequently rejected on `/api/auth/refresh`.

## Scenario 6 — `/api/auth/register` is gone

1. `POST /api/auth/register` → 404 (route removed, research.md R10).

## Automated coverage

All six scenarios above should have a corresponding TestContainers-backed integration test (constitution Principle V) — see `tasks.md` (generated by `/speckit-tasks`) for the concrete task breakdown. `AuthEndpointTests.cs` is rewritten to seed accounts via `OrganisationOnboardingWebAppFactory`'s existing director-onboarding flow plus directly-inserted rows for Staff/Parent test scenarios (no `/register` endpoint remains to seed through).
