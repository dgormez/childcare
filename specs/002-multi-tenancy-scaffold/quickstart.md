# Quickstart: Multi-Tenancy Scaffold

## Prerequisites

- Docker Postgres running locally (`childcare-postgres` container, per existing dev setup)
- Two organisations already registered via feature 001's `POST /api/organisations/register` (call it twice with two different invitations) — call them Org A and Org B, each with a director JWT
- `dotnet build backend/ChildCare.sln` succeeds

## Scenario 1 — Isolation (User Story 1, SC-001)

1. As Org A's director, call any tenant-scoped endpoint (once one exists — for this feature, `GET /health`-adjacent smoke test is insufficient since `/health` is exempt; use a minimal authenticated round-trip such as `POST /api/auth/resend-verification` once R7's shim JWT is in hand) using Org A's JWT.
2. Confirm the request resolves against Org A's own schema (inspect via `SET search_path` logs, or query Org A's schema directly to see the effect).
3. Repeat as Org B's director with Org B's JWT, concurrently (`Task.WhenAll` style, mirroring feature 001's `OrganisationOnboardingResilienceTests.cs` concurrency pattern).
4. Expected: each request only ever touches its own organisation's schema — never the other's.

## Scenario 2 — Fail closed (User Story 2, SC-002)

1. Call a non-exempt endpoint with a JWT that has no `tenant_id` claim → expect rejection before any data access, `errorKey: errors.tenant.missing`.
2. Call it with a syntactically valid but unknown `tenant_id` → expect rejection, `errorKey: errors.tenant.not_found`.
3. Seed a `Tenant` row with `ProvisioningStatus = Provisioning` (never completed) and call with its id → expect rejection, `errorKey: errors.tenant.not_ready`.

## Scenario 3 — Migration rollout (User Story 3, SC-003/SC-004)

1. Add a trivial new column to `TenantUser` (temporary, for this validation only) and generate a migration.
2. Run `dotnet run --project backend/ChildCare.Api -- migrate-tenants` (contracts/migrate-tenants-cli.md).
3. Confirm both Org A's and Org B's schemas now have the new column, with no manual per-tenant step.
4. Run the same command again — confirm it's a no-op (no error, nothing re-applied).

## Scenario 4 — Legacy auth still works (research.md R7)

1. Call `POST /api/auth/register` (no JWT) — confirm it succeeds and the returned access token, when decoded, carries a `tenant_id` claim pointing at the shim default tenant.
2. Call `POST /api/auth/resend-verification` using that token — confirm it resolves through the ordinary `TenantMiddleware` path (not the exemption list) and succeeds.

## Scenario 5 — Legacy template code is gone

1. Confirm `GET /api/habits` (or any prior Habit route), `/api/payments/*`, and `/api/notifications/push-token` return 404 — the endpoints no longer exist.
2. Confirm the solution builds with `ChildCare.Api/Data/AppDbContext.cs` and `ChildCare.Api/Models/{User,Habit,HabitCompletion,UserRefreshToken}.cs` absent.
