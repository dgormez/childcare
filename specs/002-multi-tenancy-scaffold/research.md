# Phase 0 Research: Multi-Tenancy Scaffold

Each decision below resolves a technical unknown from the Technical Context. Format: Decision / Rationale / Alternatives considered.

## R1. Reuse the existing `TenantDbContext` class as-is; add a request-scoped registration alongside the existing manual-construction usage

**Decision**: `TenantDbContext` (introduced in feature 001, `ChildCare.Infrastructure/Persistence/TenantDbContext.cs`) already takes `schemaName` as a constructor parameter and calls `HasDefaultSchema(SchemaName)` in `OnModelCreating`, using the existing `DynamicSchemaModelCacheKeyFactory` so EF's model cache is correctly keyed per schema. No second constructor or "mode" is needed. Introduce a new `ITenantDbContextResolver` (port in `ChildCare.Application.Common`, implemented in `ChildCare.Infrastructure`) with one method, `TenantDbContext ForSchema(string schemaName)`, that builds the `DbContextOptionsBuilder`/`DynamicSchemaModelCacheKeyFactory` wiring in one place. Register `TenantDbContext` itself as Scoped via `AddScoped<TenantDbContext>(sp => resolver.ForSchema(sp.GetRequiredService<ICurrentTenantService>().SchemaName))` for the request pipeline.

**Rationale**: `TenantProvisioningService` (feature 001) never resolves `TenantDbContext` through the DI container ŌĆö it constructs its own instances manually against a placeholder-then-substituted schema (research.md R6/R15 in feature 001). Registering a *separate*, DI-resolved `TenantDbContext` for request-scoped use is therefore purely additive and cannot collide with provisioning's existing manual construction. Centralizing the construction logic in `ITenantDbContextResolver` means `AuthService`'s transitional shim (R7) and the request pipeline's normal resolution both go through one reviewed code path instead of two subtly different copies.

**Alternatives considered**: A second `TenantDbContext`-like class dedicated to request-time use ŌĆö rejected, it would duplicate `OnModelCreating`/entity configuration for no benefit, since the existing class already parameterizes schema name correctly.

## R2. `ICurrentTenantService` ŌĆö Scoped, settable concrete class behind a read-only interface

**Decision**: `ICurrentTenantService` exposes only getters (`TenantId`, `SchemaName`, `TenantSlug`). The concrete `CurrentTenantService` class has public setters, registered once as `AddScoped<CurrentTenantService>()` plus `AddScoped<ICurrentTenantService>(sp => sp.GetRequiredService<CurrentTenantService>())` ŌĆö both resolve to the *same* scoped instance per request. `TenantMiddleware` resolves the concrete `CurrentTenantService` (to set it); every downstream handler/repository depends on the `ICurrentTenantService` interface (read-only).

**Rationale**: BACKLOG.md/constitution Principle I both state this MUST be scoped, never singleton ŌĆö a singleton would leak one request's tenant into a concurrent request's, which is exactly the class of bug Principle I exists to prevent. This is a different service from `TenantProvisioningService` (which *is* correctly registered Singleton in feature 001, since it holds no per-tenant state ŌĆö it opens a fresh connection and builds a fresh `TenantDbContext` on every call). The two must not be confused: one is a per-request accessor for already-resolved tenant identity, the other is a stateless provisioning tool that operates *before* any tenant context exists.

**Alternatives considered**: `AsyncLocal`-based ambient context (avoids DI plumbing) ŌĆö rejected, ASP.NET Core's built-in per-request DI scope already gives exactly the isolation needed, with no risk of ambient state leaking across async continuations onto the wrong logical request.

## R3. `TenantMiddleware` ŌĆö custom middleware after authentication, deny-by-default with an explicit exemption list

**Decision**: `TenantMiddleware` implements `IMiddleware` (not the conventional constructor+`InvokeAsync` pattern) and is registered as `AddSingleton<TenantMiddleware>()`, then wired via `app.UseMiddleware<TenantMiddleware>()` after `app.UseAuthentication()`/`app.UseAuthorization()`. This distinction matters: the conventional pattern is constructed once via `ActivatorUtilities` at pipeline-build time and is **not** resolvable from the DI container even if separately registered ŌĆö a test could never get a handle on "the" running instance. `IMiddleware`-typed middleware, by contrast, is resolved through the framework's built-in `IMiddlewareFactory` via `GetRequiredService<TenantMiddleware>()`, so the exact same singleton instance the pipeline uses is also resolvable by test code (`factory.Services.GetRequiredService<TenantMiddleware>()`), the same way feature 001's `OrganisationOnboardingResilienceTests.cs` resolves the exact `TenantProvisioningService` instance a request will use. Because the middleware itself is a singleton, its `InvokeAsync(HttpContext context, RequestDelegate next)` resolves the *scoped* `PublicDbContext`/`CurrentTenantService` from `context.RequestServices` (never via constructor injection, which would incorrectly capture one request's scoped instances for the singleton's whole lifetime). It first checks `context.GetEndpoint()?.Metadata.GetMetadata<TenantExemptAttribute>()`; if present, calls `next()` immediately with no tenant resolution. Otherwise: read the `tenant_id` claim; if absent, reject (FR-006). Look up the tenant by id in `PublicDbContext`; if not found, reject (FR-007); if found but `ProvisioningStatus != Ready`, reject (FR-008); if the lookup itself throws, reject the same way after logging the real exception server-side (FR-008a). On success, populate `CurrentTenantService` and call `next()`. `TenantMiddleware` also exposes a settable `FailureInjectionHookForTests` property (mirroring `TenantProvisioningService`'s existing test seam from feature 001), checked immediately before the `PublicDbContext` lookup, letting integration tests deterministically simulate a lookup failure (FR-008a) without needing to break the real database connection.

Named exemption list (`[TenantExempt]` applied at route registration), reflecting FR-015 plus the routes discovered while migrating the legacy auth flow (R7):

- `GET /health`
- `POST /api/admin/invitations` (feature 001, gated by a separate super-admin auth scheme)
- `POST /api/organisations/register` (feature 001, unauthenticated ŌĆö issues the *first* tenant-bearing JWT)
- `POST /api/auth/register`, `POST /api/auth/login`, `POST /api/auth/google`, `POST /api/auth/apple`, `POST /api/auth/refresh`, `POST /api/auth/forgot-password`, `POST /api/auth/reset-password`, `POST /api/auth/verify-email` ŌĆö all run *before* a tenant-bearing session exists

`POST /api/auth/logout`, `DELETE /api/auth/account`, and `POST /api/auth/resend-verification` are **not** exempt ŌĆö once R7's transitional shim is in place, the JWT these routes receive already carries a `tenant_id` claim (the shim's default tenant), so they resolve through the normal path like any other authenticated endpoint.

**Rationale**: FR-015 (clarified) requires deny-by-default with a small, explicit, visible exemption list ŌĆö the opposite of an opt-in allow-list, so a future domain-data endpoint is tenant-scoped unless someone deliberately opts it out, not the other way around. A `[TenantExempt]` marker attribute checked via endpoint metadata is the standard ASP.NET Core pattern for this (the same mechanism `[Authorize]` itself uses), keeping the exemption list visible at each route's registration rather than buried in a separate middleware allow-list that's easy to forget to update.

**Alternatives considered**: A hardcoded path-string allow-list inside the middleware itself ŌĆö rejected, it separates the exemption decision from the route definition, making it easy for a new route to be silently exempt-by-default if the middleware's list isn't updated (or, worse, silently non-exempt when it should be, breaking a legitimately tenant-agnostic endpoint). Metadata-based exemption keeps the decision colocated with the route.

## R4. Legacy `AppDbContext`-dependent template code ŌĆö delete Habits/Payments/Notifications outright; migrate Auth onto the relocated model

**Decision**: `HabitEndpoints.cs`, `PaymentEndpoints.cs` (+ `StripeService.cs`, now unused), and `NotificationEndpoints.cs` are deleted entirely, along with their DI registrations and the `Stripe.StripeConfiguration.ApiKey` wiring in `Program.cs` (the `Stripe:SecretKey`/Terraform secret itself is left in place, unused for now ŌĆö harmless, and reusable when organisation-level billing is designed later, per BACKLOG.md's invoicing feature). `AuthEndpoints.cs`/`AuthService.cs` are kept and migrated onto the relocated `TenantUser` model (R6/R7) instead of `AppDbContext.Users`. `AppDbContext.cs` itself, and `ChildCare.Api/Models/{User,Habit,HabitCompletion,UserRefreshToken}.cs`, are deleted.

**Rationale**: `PaymentEndpoints.cs`/`NotificationEndpoints.cs` were discovered during planning to also depend directly on `AppDbContext`/`Models.User` (per-user Stripe subscription, Expo push token) ŌĆö neither appears anywhere in PROJECT-BRIEF.md's Core Entities or BACKLOG.md's feature list; both are generic SaaS-template boilerplate predating the real ChildCare domain, confirmed by the user as never having been part of a tested, working flow. `AuthEndpoints.cs`, by contrast, is explicitly *not* marked "to be replaced" in PROJECT-BRIEF.md's "What Already Exists" section (unlike `AppDbContext`/`HabitEndpoints.cs`, which are), and BACKLOG.md's feature 003 ŌĆö Auth section explicitly describes evolving it further ("a working auth skeleton already exists... this spec describes what to keep, what to change, and what to add") rather than replacing it.

**Alternatives considered**: Keep Payment/Notification endpoints alive against some interim model ŌĆö rejected per user direction; there is no ChildCare domain concept (individual user billing) for them to attach to, and organisation-level billing will be designed properly, scoped correctly, when BACKLOG.md's invoicing feature is built.

## R5. `TenantUser` gains the fields `AuthService` needs; a new `TenantUserRefreshToken` entity joins it in `TenantDbContext`

**Decision**: Extend `ChildCare.Domain.Entities.TenantUser` (currently just `Id, Email, PasswordHash, Name, CreatedAt`, added in feature 001 as the provisioning baseline) with the fields carried by the legacy `ChildCare.Api.Models.User`: `GoogleId`, `AppleId`, `EmailVerified`, `EmailVerificationToken`, `EmailVerificationExpiry`, `PasswordResetToken`, `PasswordResetExpiry`. Add a new `TenantUserRefreshToken` entity (moved from `ChildCare.Api.Models.UserRefreshToken`), FK'd to `TenantUser`, also living in `TenantDbContext`. `ExpoPushToken` and the Stripe fields are **not** carried over (R4 ŌĆö their owning features are deleted).

**Rationale**: `TenantUser` and the legacy `User` both represent the same concept ŌĆö "an account inside one organisation's schema" ŌĆö so extending the existing entity avoids two parallel, confusing notions of "a tenant's user." This is exactly the target shape BACKLOG.md's feature 003 already describes ("Users table moves from the old AppDbContext into TenantDbContext... each tenant schema has its own users table"); this feature does the structural move, 003 adds the tenant-aware login *resolution* logic on top.

**Alternatives considered**: A separate new entity for auth-only fields, joined 1:1 with `TenantUser` ŌĆö rejected as needless indirection; every field here is intrinsic to "a user account," not a separable concern.

## R6. Baseline tenant migration gains the extended `users` table + `refresh_tokens` table

**Decision**: The existing `InitialTenantSchema` migration (feature 001, `ChildCare.Infrastructure/Persistence/Migrations/Tenant/`) is superseded by a new migration reflecting the extended `TenantUser` shape (R5) plus the new `TenantUserRefreshToken` table. `TenantProvisioningService`'s runtime script-generation approach (feature 001 research.md R15) is unaffected ŌĆö it always generates the script fresh from whatever migrations exist, so a brand-new tenant provisioned after this feature ships gets the extended baseline automatically. Existing already-provisioned tenants (from feature 001's testing) pick up the new tables via the rollout mechanism (R8).

**Rationale**: Keeps a single source of truth for "what a brand-new tenant schema looks like" (the compiled migrations), consistent with feature 001's design ŌĆö no parallel, hand-maintained baseline-SQL definition to drift out of sync.

**Alternatives considered**: A second, separate migration applied only to *new* tenants going forward, leaving already-provisioned tenants on the old shape until manually reconciled ŌĆö rejected, this is precisely the scenario the rollout mechanism (User Story 3) exists to avoid needing.

## R7. Transitional "default tenant" shim for pre-authentication routes

**Decision**: Until feature 003 delivers the real emailŌåÆorganisation lookup, the pre-authentication `AuthEndpoints.cs` routes (register, login, google, apple, refresh, forgot-password, reset-password, verify-email ŌĆö the exemption list in R3) resolve a single **default tenant** ŌĆö the earliest-created `Ready` tenant in `PublicDbContext.Tenants` ŌĆö via a small internal helper, and use `ITenantDbContextResolver.ForSchema(...)` (R1) directly to obtain a `TenantDbContext` for that schema, bypassing `ICurrentTenantService`/`TenantMiddleware` entirely (since no tenant is resolved yet at that point in the pipeline). Critically, `JwtService`'s existing tenant-aware overload (`GenerateAccessToken(Guid userId, string email, Guid tenantId)`, added in feature 001) is used to issue the login-success JWT ŌĆö with the shim's default tenant's id ŌĆö instead of the old `GenerateAccessToken(User user)` overload (removed, since `Models.User` is deleted). This means every JWT the legacy auth flow issues *does* carry a real `tenant_id` claim, so the **non**-exempt, post-login routes (`logout`, `account` deletion, `resend-verification`) resolve through the completely ordinary `TenantMiddleware` path with no special-casing at all.

**Rationale**: This is the one deliberately temporary piece of this feature, flagged as such in code comments and here. It satisfies the user's chosen resolution to the FR-014 clarification ŌĆö auth endpoints keep compiling and running end-to-end ŌĆö without pretending to solve real multi-tenant login early (there is no way to know which of several organisations an email belongs to without the index feature 003 delivers). Because the shim only decides *which* tenant to use at the few pre-auth entry points, and every other authenticated route resolves normally from the JWT's `tenant_id` claim, the amount of code feature 003 needs to change is small and well-contained: swap the "pick the default tenant" helper for a real emailŌåÆtenant_id lookup, and the rest of the pipeline (`TenantMiddleware`, `ICurrentTenantService`, `TenantDbContext` resolution) is untouched.

**Important implementation constraint**: `AuthService` MUST NOT take the DI-registered, Scoped `TenantDbContext` (R1/R13) as a constructor dependency ŌĆö for the pre-auth exempt routes, `TenantMiddleware` never runs, so `ICurrentTenantService` (which that registration reads `SchemaName` from) is still unset at the point DI would construct it, silently producing a `TenantDbContext` pointed at an empty/wrong schema. `AuthService` must instead call `ITenantDbContextResolver.ForSchema(...)` explicitly in every method, choosing the schema itself: the shim's default-tenant lookup for the exempt pre-auth routes, or `ICurrentTenantService.SchemaName` (already correctly set by `TenantMiddleware`, since these routes are non-exempt) for `logout`/`account`/`resend-verification`. One consistent internal pattern across all of `AuthService`'s methods, rather than some methods relying on DI-injected `TenantDbContext` and others not.

**Alternatives considered**: Deleting `AuthEndpoints.cs`/`AuthService.cs` entirely for this feature (mirroring R4's treatment of Habits/Payments/Notifications) ŌĆö rejected per explicit user direction (Option B) and PROJECT-BRIEF.md/BACKLOG.md's signal that Auth is meant to evolve, not be thrown away.

## R8. Migration rollout mechanism ŌĆö a CLI subcommand in `ChildCare.Api`, not a startup utility, not a new project

**Decision**: A CLI subcommand (e.g., `dotnet run -- migrate-tenants`, checked at the very top of `Program.cs` before the normal web-host build path, exiting after completion) iterates every tenant in `PublicDbContext` and applies pending `TenantDbContext` migrations to each one's schema via `context.Database.MigrateAsync()`, using `ITenantDbContextResolver.ForSchema(...)` (R1) to build the per-tenant context. No new project is introduced.

**Rationale**: Constitution Principle VI's migrations-MUST-NOT-auto-apply rule has exactly one carve-out (brand-new, empty tenant schemas at onboarding time, feature 001) and explicitly states rolling reviewed migrations out to *existing* tenant schemas "remains a deliberate, explicit operation, not something any feature auto-applies blindly" ŌĆö ruling out a startup-time auto-apply utility for this purpose. A separate console project would be a 6th project with no documented justification, violating Principle VII (Monolith-First); a subcommand inside the existing `ChildCare.Api` entry point avoids that while still being a single, explicit, operator-triggered action (satisfying FR-010/FR-011/SC-003/SC-004).

**Alternatives considered**: A startup utility gated behind a config flag ("only runs if explicitly enabled") ŌĆö rejected as functionally identical in risk to an unconditional auto-apply (a flag left on by accident in production has the same blast radius the constitution's rule exists to prevent); a dedicated 6th project ŌĆö rejected per Principle VII, no need justifies the extra project here.

## R9. Orphaned legacy tables ŌĆö manual, reviewed cleanup script, not an automatic migration

**Decision**: The tables `Users`, `Habits`, `HabitCompletions`, `UserRefreshTokens` (owned by the now-deleted `AppDbContext`, unqualified/default Postgres schema) are left in place after this feature ships, to be dropped via a manual, reviewed SQL script run once per environment (local dev, Neon) ŌĆö not generated by an EF migration, since no `DbContext` models them anymore for EF to generate a `Down()` from.

**Rationale**: Consistent with constitution Principle VI's default (migrations/schema changes against existing, populated-or-not shared structures are deliberate, reviewed, manual operations) ŌĆö this is exactly the "existing schema" case the rule is written for, not the onboarding carve-out. The tables are otherwise inert (nothing references them once `AppDbContext` is gone) so leaving them briefly is zero-risk.

**Alternatives considered**: Auto-dropping them via a startup check ŌĆö rejected, same reasoning as R8; irreversible `DROP TABLE` statements are exactly the kind of operation that must stay a deliberate, human-reviewed action.

## R10. i18n error keys for the new rejection paths

**Decision**: Following `backend/ERROR_KEYS.md`'s existing convention (feature 001): `errors.tenant.missing` (FR-006, 401), `errors.tenant.not_found` (FR-007/FR-008a-unknown-case, 403), `errors.tenant.not_ready` (FR-008, 403) ŌĆö note FR-008a requires the *unknown* and *lookup-failed* cases to be indistinguishable to the caller, so a lookup failure also returns `errors.tenant.not_found` (the true cause is logged server-side via the existing global exception handler pattern, not returned to the client).

**Rationale**: Constitution Principle IV (NON-NEGOTIABLE) ŌĆö every error response uses a locale-aware key. Reusing the established `errors.<area>.<reason>` naming convention from `ERROR_KEYS.md` keeps this consistent with feature 001 rather than inventing a new scheme.

**Alternatives considered**: A single generic `errors.tenant.rejected` for all three FR-006/007/008 cases ŌĆö rejected; FR-006 (missing claim) and FR-008 (not-ready tenant) are legitimately distinguishable to the caller per the spec (only the unknown-vs-lookup-failure pair, FR-008a, must be indistinguishable), so collapsing all three would lose information the spec explicitly permits keeping.

## R11. Connection mode ŌĆö unchanged from feature 001

**Decision**: Continue using Neon's direct (non-pooled) connection string in deployed environments; no pgBouncer transaction-mode pooling. No new decision needed ŌĆö this is the same constraint feature 001 already established and this feature's `TenantDbContext`/`ITenantDbContextResolver` reuse the same `ConnectionStrings:DefaultConnection` configuration.

**Rationale**: FR-012; constitution Principle I. Restated here only for completeness of the Technical Context, not because anything changes.
