# Quickstart: Staff Management

## Prerequisites

- Docker Postgres running locally
- Two organisations registered via feature 001's `POST /api/organisations/register` (as in features 002–004's quickstarts) — Org A (slug `org-a`) and Org B (slug `org-b`), each with a director account and a valid access token
- Org A has at least one location created via feature 004's `POST /api/locations`
- `dotnet build backend/ChildCare.sln` succeeds

## Scenario 1 — Director creates a staff profile and the invitee logs in (User Story 1, SC-001/SC-002/SC-003)

1. As Org A's director, `POST /api/staff` with `firstName`/`lastName`/`email`/`phone`/`qualificationLevel: "QualifiedCaregiver"`/`role: "Staff"` → `201`, response includes a new `id`; an invitation email is sent (logged to console when SMTP isn't configured, matching feature 001/003's existing dev-mode behavior).
2. `GET /api/staff` → the new profile appears, `deactivatedAt: null`.
3. `POST /api/staff` omitting `qualificationLevel` with `role: "Staff"` → `422 errors.validation`, `fieldErrors.qualificationLevel` populated.
4. Extract the invitation token (test seam — same pattern as feature 001's invitation tests), `POST /api/staff/accept-invitation` with `token` + a new `password` → `200`.
5. Log in via feature 003's existing `POST /api/auth/login` with `organisationSlug: "org-a"`, the staff member's `email`, and the new password → `200`, JWT issued with `Role: Staff`.

## Scenario 2 — Director's own account gains an optional Staff Profile (User Story 1 edge case, Clarifications)

1. As Org A's director (account already exists from feature 001 onboarding), `POST /api/staff` with `existingTenantUserId` set to the director's own `TenantUserId`, `qualificationLevel: "QualifiedCaregiver"` → `201`, no invitation email sent (director already has credentials).
2. `GET /api/staff` → the director now appears in the roster alongside Staff-role profiles.
3. `POST /api/staff/{id}/deactivate` on the director's own Staff Profile → `200`; the director drops out of `GET /api/staff`, but their existing login (`POST /api/auth/login`) continues to work — deactivating a Director's Staff Profile never blocks that account (FR-010).

## Scenario 2a — Resend a staff invitation (FR-006a)

1. As Org A's director, create a staff profile as in Scenario 1 step 1.
2. `POST /api/staff/{id}/resend-invitation` → `200`; the original invitation token from step 1 now fails accept-invitation with `400 errors.staff.invitation_invalid_or_expired`, while the newly-issued token succeeds.
3. `POST /api/staff/{id}/resend-invitation` again after the invitee has already accepted (password set) → `409 errors.staff.account_already_active`.

## Scenario 3 — Multi-location eligibility (User Story 2, SC-005)

1. Create a second location in Org A via feature 004's `POST /api/locations`.
2. `PUT /api/staff/{id}/locations/{locationAId}` then `PUT /api/staff/{id}/locations/{locationBId}` → both `200`; `GET /api/staff/{id}` shows both in `eligibleLocationIds`.
3. `DELETE /api/staff/{id}/locations/{locationAId}` → `200`; `GET /api/staff/{id}` shows only location B.
4. A staff profile with zero eligible locations (never assigned) → `GET /api/staff/{id}` shows an empty `eligibleLocationIds` array, no error.

## Scenario 4 — Update profile fields and photo (User Story 3)

1. `PUT /api/staff/{id}` changing `phone`/`qualificationLevel` → `200`, `GET /api/staff/{id}` reflects the change.
2. `POST /api/staff/{id}/photo/upload-url` → `200`, response includes `uploadUrl`/`objectPath`.
3. `PUT` an image directly to `uploadUrl` (outside the API, direct-to-GCS).
4. `GET /api/staff/{id}` → `photoDownloadUrl` is populated with a freshly-signed URL that successfully retrieves the uploaded image.

## Scenario 5 — Deactivate and reactivate (User Story 4, SC-004)

1. `POST /api/staff/{id}/deactivate` on a profile with no dependents → `200`, `deactivatedAt` set.
2. `GET /api/staff` (default) → the deactivated profile is absent.
3. Attempt login as the deactivated staff member (feature 003's `POST /api/auth/login`) → login fails (deactivated accounts cannot authenticate).
4. `POST /api/staff/{id}/reactivate` → `200`, `deactivatedAt` cleared; login succeeds again.

## Scenario 6 — Tenant isolation (constitution Principle I, FR-015)

1. Note a staff profile `id` created in Org A (Scenario 1).
2. As Org B's director (different access token, different `tenant_id` claim), `GET /api/staff/{orgAStaffId}` → `404 errors.staff.not_found` (not `403` — structurally invisible via `search_path`).
3. `GET /api/staff` as Org B → Org A's staff never appear.
