# Phase 0 Research: Staff Management

Each decision below resolves a technical unknown from the Technical Context. Format: Decision / Rationale / Alternatives considered.

## R1. `StaffProfile` is a new entity linked 1:1 to `TenantUser`, not fields bolted onto `TenantUser` itself

**Decision**: `StaffProfile` is a separate entity with a required, unique foreign key `TenantUserId` back to the existing `TenantUser` (feature 003). It holds `FirstName`, `LastName`, `Phone`, `QualificationLevel` (nullable enum), `ProfilePhotoObjectPath` (nullable string — the GCS object path, not a URL, see R3), and `DeactivatedAt` (nullable, mirrors `Location`'s soft-delete shape from feature 004). It carries no `OrganisationId`/tenant column — same structural reasoning as `TenantUser`/`Location` (schema is the isolation boundary, constitution Principle I).

**Rationale**: `TenantUser` today holds only account/auth concerns (email, password hash, OAuth ids, role, verification tokens) shared across all three roles (Director/Staff/Parent). Adding caregiver-specific fields (qualification, phone, photo) directly onto `TenantUser` would mean every `Parent`-role row also carries unused caregiver columns, and would conflate "the account" with "the operational profile" — two things this spec's Clarifications session deliberately treats as separable (a `Director` account gains a Staff Profile only if the director opts in to appearing in caregiver rosters). A separate 1:1 entity models that optionality cleanly: no `StaffProfile` row exists for an account that hasn't opted in, versus a `TenantUser` with a pile of nullable caregiver columns for every Parent account.

**Alternatives considered**: Extending `TenantUser` in place with nullable caregiver columns — rejected, since it pollutes the account entity with role-specific data for the two roles that don't need it (Parent always, Director unless opted-in) and makes "does this person have a staff profile" a set of null-checks across five columns instead of one row's existence.

## R2. `StaffInvitation` is a new, tenant-scoped entity — not a reuse of the public-schema `Invitation`

**Decision**: `StaffInvitation` lives in `ITenantDbContext`/`TenantDbContext` (tenant schema), not `IPublicDbContext`. Shape mirrors `Invitation` (feature 001): `Id`, `Email`, `TokenHash` (byte[]), `ExpiresAt`, `CreatedAt`, plus a required `StaffProfileId` FK (the profile the invitation is provisioning credentials for) — no `UsedAt` column, "used" is derived from whether the linked `TenantUser.PasswordHash` has been set (mirrors feature 001's R10 "used is derived, not stored" pattern). `InvitationTokenCodec` (feature 001, `ChildCare.Application/Invitations/`) is reused as-is for token generation/hashing — no new crypto code.

**Rationale**: Feature 001's `Invitation` exists in the public schema because its entire purpose is provisioning a *new tenant* before any tenant schema exists to put it in. A staff invitation is the opposite case: the tenant schema already exists, and the invitation's entire purpose is scoping to *that one* tenant (an invited email must resolve unambiguously to one organisation's staff roster, and must never leak across schemas). Reusing the *codec* but not the *entity* follows the direction flagged in feature 003's shipped notes ("assumed they'll reuse the same `TenantUser`-creation primitives") — the primitive being reused is the token-generation mechanism, not the storage location, since the storage location has a structurally different scoping requirement.

**Alternatives considered**: Reusing the public-schema `Invitation` entity with an added tenant-identifying field — rejected, since it would mean a tenant-scoped concept living in the one schema explicitly reserved for cross-tenant metadata (constitution Principle I: "`PublicDbContext`... contains only the shared `tenants` table"), and would require every staff-invitation query to filter by tenant manually instead of getting it for free from `search_path`.

## R3. Profile photos — GCS V4 signed URLs via a new `IProfilePhotoStorage` port, object path stored (not a URL)

**Decision**: Add `ChildCare.Application.Common.IProfilePhotoStorage` with two methods: `Task<(string objectPath, string uploadUrl)> CreateUploadUrlAsync(Guid staffProfileId)` and `Task<string?> CreateDownloadUrlAsync(string? objectPath)`. `StaffProfile.ProfilePhotoObjectPath` stores only the GCS object path (e.g. `tenants/{schema}/staff/{staffProfileId}/photo.jpg`) — never a URL. A signed *download* URL is generated fresh on every read (`GetStaffByIdQuery`/`ListStaffQuery` responses), and a signed *upload* URL is generated on demand (`RequestPhotoUploadUrlCommand`) for the client to `PUT` the image bytes directly to GCS — the ASP.NET API never proxies image bytes. `GcsProfilePhotoStorage` (the concrete implementation, `ChildCare.Infrastructure/Storage/`) uses `Google.Cloud.Storage.V1`'s `UrlSigner` (V4 signing) with credentials supplied via a service-account JSON path from configuration (environment variable / Secret Manager, constitution Principle VI — never hardcoded, following the same configuration-injection pattern as Google/Apple OAuth validation in feature 003).

**Rationale**: Storing a signed URL itself (rather than the object path) would bake an expiring value into the database — it would go stale the moment its signature window elapses, silently breaking photo display until the next edit. Storing only the path and re-signing on every read keeps the URL always fresh and matches the constitution's "signed, time-limited... URLs" requirement literally (a URL that's read from the database unchanged for months is not meaningfully time-limited). Direct-to-GCS upload (client `PUT`s to a signed URL) avoids adding binary-upload handling (multipart parsing, size limits, streaming) to the API for what is fundamentally a storage-provider concern GCS already solves.

**Alternatives considered**: Proxying uploads through an API endpoint (`POST /api/staff/{id}/photo` with multipart body, API relays to GCS) — rejected as unnecessary complexity and load on the API for pure byte-shuffling; direct-to-storage upload is the standard pattern for this exact problem and needs no proxy. Storing a long-lived signed URL (rather than re-signing per read) — rejected, since GCS V4 signed URLs cap out at 7 days validity by design, which is not "time-limited" in any meaningful sense if treated as permanent storage, and would require a background refresh job this feature has no other reason to build.

## R4. Deactivation dependent-check extension point — `IEnumerable<IStaffDeactivationGuard>`, empty for this feature

**Decision**: Add `ChildCare.Application.Common.IStaffDeactivationGuard` with `Task<bool> HasActiveDependentsAsync(Guid staffProfileId, ITenantDbContext db, CancellationToken ct)`, resolved as `IEnumerable<IStaffDeactivationGuard>` by `DeactivateStaffProfileCommandHandler` — identical shape to feature 004's `ILocationDeactivationGuard`. Zero implementations are registered by this feature (no `services.AddScoped<IStaffDeactivationGuard, ...>()` call exists yet); with nothing registered, the `IEnumerable` resolves empty and deactivation always succeeds, matching spec.md FR-011/Assumptions.

**Rationale**: Directly reuses the pattern feature 004 established (and which this spec's FR-011 explicitly asks to mirror) rather than inventing a new extension mechanism — features 009 (attendance/scheduling dependents) and 011 (caregiver scheduling) can each register their own guard independently later without one overwriting the other, for the same multiple-independent-registrations reason `ILocationDeactivationGuard` uses `IEnumerable<T>` instead of a single swappable implementation.

**Alternatives considered**: None seriously considered beyond the established pattern — inventing a different mechanism for what is structurally the same problem feature 004 already solved would be inconsistent for no benefit.

## R5. Staff account creation is transactional: `TenantUser` + `StaffProfile` + `StaffInvitation` in one command

**Decision**: `CreateStaffProfileCommandHandler` creates all three rows (`TenantUser` with `Role = Staff`, empty `PasswordHash` until the invitation is accepted; `StaffProfile`; `StaffInvitation`) inside a single `SaveChangesAsync()` call (one EF Core `DbContext`, one transaction). `AcceptStaffInvitationCommandHandler` validates the token against `StaffInvitation.TokenHash`/`ExpiresAt`, sets `TenantUser.PasswordHash`, and marks the invitation implicitly used (R2 — no explicit `UsedAt`, derived from `PasswordHash` being non-empty).

**Rationale**: A `TenantUser` with no corresponding `StaffProfile`, or a `StaffProfile` with no backing `TenantUser`, would be an invalid intermediate state with no valid use case (spec.md FR-001/FR-005 always create them together for a new Staff account) — a single transaction makes that state unreachable rather than merely unlikely. This mirrors feature 001's registration handler, which similarly creates a `Tenant` + tenant schema + `TenantUser` together rather than as separable steps.

**Alternatives considered**: Two-step creation (create `TenantUser` first, `StaffProfile` as a follow-up call) — rejected, since it introduces a window where a bare `TenantUser` with `Role = Staff` exists with no profile, which every other part of this spec (rosters, eligibility, BKR qualification) assumes cannot happen.

## R6. Existing Director account gaining a `StaffProfile` reuses the same creation handler, invitation step skipped

**Decision**: `CreateStaffProfileCommand` accepts an optional `ExistingTenantUserId` parameter. When supplied (and the referenced `TenantUser.Role == Director`), the handler creates only a `StaffProfile` row linked to that existing `TenantUserId` — no new `TenantUser`, no `StaffInvitation`, no email sent, since the director already has working credentials (feature 001/003). When absent, the handler follows the full R5 path (new `TenantUser` + `StaffProfile` + `StaffInvitation`).

**Rationale**: Directly implements the Clarifications session's resolution that a Director account may gain an optional Staff Profile without a second account. Branching within one command (rather than a wholly separate "attach staff profile to director" command) keeps the "does this profile creation also need to provision login credentials" decision in one place, matching FR-001's framing that a Staff Profile "MAY also be attached to an existing Director account."

**Alternatives considered**: A second, separate command (`AttachStaffProfileToDirectorCommand`) — rejected as an unnecessary second entry point for what is, from the data-model's perspective, the same `StaffProfile`-row creation with one fewer side effect; the branch is small enough (skip two of three row creations, skip the email) not to justify a second command/validator/handler trio.

## R7. Qualification level is a nullable enum, conditionally required by the validator (not the schema)

**Decision**: `StaffProfile.QualificationLevel` is `QualificationLevel?` (nullable enum: `QualifiedCaregiver`, `Auxiliary`, `StudentVolunteer`) at the database level — always nullable in the schema, regardless of role. `CreateStaffProfileCommandValidator`/`UpdateStaffProfileCommandValidator` enforce the conditional rule from FR-003 in application code: required (`.NotNull()`) when the target `TenantUser.Role == Staff`, optional when `Role == Director`.

**Rationale**: A database-level `NOT NULL` constraint can't express "required for one role, optional for another" without a `CHECK` constraint referencing a join, which EF Core doesn't model cleanly and would be harder to evolve than a FluentValidation rule. Enforcing the conditional requirement in the validator (constitution Principle III: "FluentValidation... fires before every command handler") keeps the business rule in the Application layer where every other cross-field rule in this codebase already lives (e.g., feature 004's phone-format/max-length rules), rather than splitting it between a DB constraint and application code.

**Alternatives considered**: A `CHECK` constraint enforcing the conditional nullability at the database level — rejected as unnecessary defense-in-depth for a rule with no security/tenant-isolation stakes (unlike, say, cross-tenant leakage); the validator is the single source of truth for this rule, consistent with how every other conditional field-requirement in this codebase (e.g., `dossiernummer`/`verantwoordelijke` nullability in feature 004) is enforced only in the Application layer.

## R8. Read side — `ListStaffQuery`/`GetStaffByIdQuery` as MediatR queries, eligible-locations eager-loaded

**Decision**: Both reads are MediatR queries in `ChildCare.Application/Staff/`, consistent with feature 004's R6 choice. `GetStaffByIdQuery`/`ListStaffQuery` results include the staff member's eligible locations (joined from `StaffLocationEligibility`) and a freshly-signed photo download URL (R3) inline in the response DTO — no separate "get eligible locations" round-trip is needed for the common case of viewing a staff profile.

**Rationale**: Consistency with feature 004's established read-side pattern (MediatR for all reads, not just writes). Inlining eligible locations avoids an N+1 pattern on the staff list view (a director viewing the roster needs to see each person's locations without a follow-up call per row).

**Alternatives considered**: A separate `GetStaffLocationEligibilityQuery` — rejected as an unnecessary extra round-trip for data that's always wanted alongside the base profile in this feature's UI flows.

## R9. Follow-ups from `/speckit-checklist` and `/speckit-analyze` — all fixed, none left as accepted debt

Per project policy, every finding raised during the requirements-quality checklist pass and the cross-artifact analyze pass was fixed rather than logged as advisory. Three are substantial enough to record here as research decisions in their own right:

- **Concurrent duplicate-email invitations (checklist CHK020)**: `CreateStaffProfileCommandHandler` (T032) wraps `SaveChangesAsync` in a try/catch for a unique-constraint `DbUpdateException` on `Users.Email`, mapping it to `StaffResult.EmailAlreadyExists` — the exact `IsUniqueConstraintViolation(DbException { SqlState: "23505" })` pattern `RegisterOrganisationCommandHandler` (feature 001) already established, reused rather than re-invented.
- **Invitation single-use enforced before expiry, not just after (checklist CHK021)**: `AcceptStaffInvitationCommandHandler` (T034) checks `TenantUser.PasswordHash` is still empty *before* accepting a token that otherwise still passes `ExpiresAt` — a second accept attempt on an already-used-but-not-yet-expired token is rejected identically to an expired one (FR-006b). Without this check, R2's "used is derived from PasswordHash" design would silently allow re-running accept-invitation to reset a password an unlimited number of times within the expiry window, which is a materially different (weaker) guarantee than "single-use."
- **Deactivation must not silently leave a live session usable, and must never apply to a Director (analyze finding C1 area / checklist CHK006, CHK016)**: `DeactivateStaffProfileCommandHandler` (T058, amended by T071) removes all of the linked `TenantUser`'s `TenantUserRefreshToken` rows when deactivating a `Staff`-role profile, mirroring `ResetPasswordCommandHandler`'s existing session-invalidation. `LoginCommandHandler`'s deactivation check (T059) is conditioned on `TenantUser.Role == Staff` specifically — a `Director` account's own optional Staff Profile being deactivated must never block that director's login, since the Director account's authority does not derive from the Staff Profile at all (FR-002).

No finding from either pass was dismissed as "acceptable precedent" without an accompanying fix — see `checklists/requirements-quality.md`'s per-item resolution notes for the full list.
